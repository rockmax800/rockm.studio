# 30 — Delivery Lane

> Layer 1 — Core Engine
>
> **Always active.** Controls the deterministic release pipeline from code to production.

## 1 — Purpose

Defines the single delivery pipeline that enforces a deterministic, auditable path from code change to production deployment. Every code change must flow through this pipeline — no shortcuts. All delivery events are recorded in the canonical `event_log` (see `core/12-event-log-architecture.md`).

---

## 2 — Architecture Constraints

| Constraint | Value |
|-----------|-------|
| VCS Provider | GitHub only |
| CI Provider | GitHub Actions only |
| Deploy Target | Single VPS |
| Container Runtime | Docker |
| Auto-production deploy | **Forbidden** |
| Force merge | **Forbidden** |
| Auto domain binding | **Forbidden** |

---

## 3 — Pipeline Flow

```
Blueprint → Task → Run → Feature Branch
    │
    ▼
RepoWorkspace (created by RunService)
    │
    ▼
RunExecutor produces code → commit → push
    │
    ▼
PullRequest record (opened)
    │
    ▼
GitHub Actions CI → CheckSuite (queued → running → passed|failed)
    │
    ├── CI failed → PR blocked, cannot merge
    │
    ▼ CI passed
PR Merge → PullRequest.status = merged
    │
    ▼
Staging Deployment (deploying → live|failed)
    │
    ├── Build Docker image
    ├── Push to registry (ghcr.io)
    ├── SSH to VPS
    ├── docker pull + restart
    │
    ▼ Staging live
Founder reviews staging
    │
    ├── Rejected → no promotion, fix required
    │
    ▼ Approved
Production Deployment (deploying → live)
    │
    ├── Same Docker image promoted
    ├── SSH to VPS
    ├── docker pull + restart production container
    │
    ▼
Live in production
```

---

## 4 — Entity Relationships

```
Repository (GitHub)
  └── RepoWorkspace (per Run)
        └── PullRequest (per validated Task)
              └── CheckSuite (GitHub Actions)
                    └── Deployment (staging → production)
```

All entities traceable: `Project → Task → Run → Workspace → PR → CI → Deploy`

### 4.1 — Artifact Evidence Chain (v3.0)

Each delivery step produces a typed artifact for full traceability:

```
Run → implementation_patch (references workspace)
  → Review → review_report (references review verdict)
    → CheckSuite → qa_evidence (references CI results)
      → Deployment → deployment_receipt (references live deployment)
        → release_note (user-facing summary)
```

See `core/11-artifact-type-system.md` for category contracts.

## 6 — PR Merge Guard

Before a PR can be merged:

1. PR.status must be `opened`
2. At least one CheckSuite must exist for the PR
3. Latest CheckSuite.status must be `passed`

If any condition fails → merge blocked via GuardError.

---

## 7 — CI Integration (GitHub Actions)

CheckSuite maps to a GitHub Actions workflow run:

| CheckSuite Field | GitHub Actions Equivalent |
|-----------------|--------------------------|
| external_run_ref | Workflow run ID |
| status: queued | Workflow queued |
| status: running | Workflow in progress |
| status: passed | Workflow completed (success) |
| status: failed | Workflow completed (failure) |
| logs_ref | Workflow logs URL |

CI is triggered by GitHub webhook on PR creation/update. CheckSuite records are created/updated by a webhook handler (future implementation).

## 8 — Staging Deployment

### Trigger
PR merged → staging deployment created automatically.

### Process
1. Create Deployment record: `environment=staging, status=deploying`
2. Build Docker image tagged with version label
3. Push to container registry (ghcr.io)
4. SSH to VPS: `docker pull && docker stop && docker rm && docker run`
5. Health check
6. Update Deployment: `status=live` or `status=failed`

### Docker Commands (generated by DeliveryLaneService)
```bash
docker pull ghcr.io/owner/repo:v1.2.3
docker stop app-staging || true
docker rm app-staging || true
docker run -d --name app-staging --restart unless-stopped ghcr.io/owner/repo:v1.2.3
```

---

## 9 — Production Gate

Production deployment is **never automatic**. Requirements:

1. Staging deployment with `status=live` must exist
2. Founder explicitly approves promotion
3. `promoteToProduction()` creates a new Deployment record

### Production Deploy
Same Docker image as staging (no rebuild). Only container name and config differ.

---

## 10 — Rollback Procedure

### Trigger
Founder initiates rollback via `DeliveryLaneService.rollback()`.

### Process
1. Validate target deployment was previously `live`
2. Mark current live deployment as `rolled_back`
3. Create new Deployment with `rollback_of_deployment_id` set
4. Deploy the target deployment's Docker image tag
5. Update new deployment status

### Rollback creates a new deployment record — it does not modify the original.

---

## 11 — Failure Scenarios

| Scenario | Response |
|----------|----------|
| CI fails | PR blocked. Fix code, push, re-run CI |
| Docker build fails | Deployment.status = failed. Fix Dockerfile, retry |
| VPS unreachable | Deployment.status = failed. Alert founder, retry when VPS recovers |
| Health check fails | Deployment.status = failed. Auto-rollback NOT supported — founder decides |
| Production regression | Founder initiates rollback to last known-good deployment |

---

## 12 — Outbox Integration

All deployment events are written to the transactional outbox:

| Event | When |
|-------|------|
| `pull_request.merged` | PR merged |
| `deployment.staging_started` | Staging deploy begins |
| `deployment.live` | Deploy succeeds |
| `deployment.failed` | Deploy fails |
| `deployment.production_started` | Production promotion |
| `deployment.rollback_started` | Rollback initiated |

---

## 13 — Safety Invariants

1. **No force merge** — CI must pass before PR merge
2. **No auto-production deploy** — founder approval always required
3. **No auto domain binding** — DNS/TLS changes are manual
4. **All deploys logged** — ActivityEvent + OutboxEvent for every deployment state change
5. **Rollback always possible** — via `rollback_of_deployment_id` chain
6. **Single VPS** — no multi-cloud, no Kubernetes, no distributed deploys

---

## 14 — Deployment Secret Governance

All deployment operations follow the secret injection rules defined in `delivery/runtime-and-secret-governance.md`.

| Deploy Step | Secrets Used | Available To |
|------------|-------------|-------------|
| Docker build | None | Execution Plane |
| Registry push | Registry credentials | Execution Plane only |
| SSH to VPS | SSH deploy key | Execution Plane only |
| Image pull on VPS | Registry credentials (pre-configured) | VPS |
| Health check | None | Public endpoint |

**Invariants:**
- Sandbox containers never have deploy credentials
- Control Plane never performs deployment operations
- Same Docker image promoted staging → production (no rebuild)
- All deployment secrets masked in logs
